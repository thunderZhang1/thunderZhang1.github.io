浅谈Android架构

发表于 2016-06-22 | 更新于 2018-10-26 | 分类于 Android

1、MVC

MVC即Model-View-Controller,MVC是目前大多数企业采用J2EE的结构设计，主要适用于交互式的Web应用。

MVC模式的结构分为三部分，实体层的Model，视图层的View，以及控制层的Controller。

  • Model层：JavaBean实体类，用于保存实例数据
  • View层：UI界面，用于显示数据和接收用户的手势和输入
  • Controller层：控制器用于更新UI界面和数据实例

在Android的开发过程中，每个层对应如下：

  • Model层：对应JavaBean、Database、SharePreference和网络请求等；
  • View层：对应xml布局、自定义View或ViewGroup；
  • Controller层：对应Activity、Fragment；

以用户点击登录按钮为例：View接收用户的点击行为，并告知controller层用户的意图，controller将用户输入的表单数据通过model层提交到服务端，model数据更新时通知controller更新View。

这种模式的优点在于Controller层起到桥梁作用，在View层和Model层之间通信，使得View层和Modle层分离解耦。但是由于在Android中View层的XML控制能力太弱，Controler层的Activity并没有和View层完全分离。当需要动态改变一个页面的显示（如背景、显示隐藏按钮等），都无法在xml中处理，只能在Activity中处理。造成了Activity即时Controller层又是View层，代码繁冗。想必你也有过Activity过于庞大的苦恼吧。

2.MVP

由于MVC在android开发中局限性，在MVC的基础上变种一种新的MVP.因为Activity既充当了Controller的角色，有包含了View的逻辑。所以将原本View的逻辑抽象成View接口，将Controller的业务逻辑抽象成Presenter接口，并将Activity归为View层，弥补XML控制View的不足，这便是MVP的由来。

这里的Presenter层持有View和Model的实例，等同于MVC中的controller层。
下面以google在github上提供的官方demo中感受下MVP的妙处吧。

    interface View extends BaseView<Presenter> {
        void setLoadingIndicator(boolean active);
        void showTasks(List<Task> tasks);
          ......
    }

    interface Presenter extends BasePresenter {
        void result(int requestCode, int resultCode);
        void loadTasks(boolean forceUpdate);
        void addNewTask();    
        ......    
    }
   }

分别继承了BaseView和BasePresenter，View和Presenter中实现的方法分别是UI操作，和数据业务逻辑操作， 此时是不是看的异常的清晰。

总结
  1. 首先BaseView和BasePresenter中定义UI和业务上都要用的方法，比如UI的showLoading(),hideLoading()等
  2. 定义Contract接口，包含View和Presenter接口分别继承BaseView和BasePresenter，并分别定义UI和业务逻辑方法。
  3. 定义XXXPresenter implements XXXContract.Presenter,包含一个XXXContract.View属性
  4. 定义XXXActivity/XXXFragment implements XXXContract.View，
  5. 将XXXActivity/XXXFragment和XXXPresenter通过构造方法或set方法等进行绑定

3、clean架构

Clean架构中，代码被分成“洋葱形”,从外向内层层依赖，内层不能依赖外层，即内层不知道有关外层的任何事情。uncle Bob的demo


Clean架构可以使你的代码有如下特性：

  • 独立于架构
  • 易于测试
  • 独立于UI
  • 独立于数据库
  • 独立于任何外部类库

一般项目会分3层：

  • domain layer: 主要定义POJO,定义业务接口
  • data layer: domain中定义接口的具体实现
  • presentation layer: UI逻辑,不处理跟UI无关的任务业务
domain层

domain中的代码不会出现任何跟UI相关的任何逻辑

  1. 定义普通的POJO类
  2. 定义Use Case:包含了App可以做的所有操作。
  3. 定义核心逻辑接口，不关心怎么实现，只为核心业务逻辑定义规则
data层

data中的代码不会出现任何跟UI相关的任何逻辑

  1. 依赖domain层，是domain层的具体实现
  2. 网络请求框架
  3. 对数据进行操作，保存，包装，映射等
presentation层

presentation层主要处理UI逻辑，不关心数据来源等跟UI无关的逻辑。
